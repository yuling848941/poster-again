## Context
当前系统使用临时文件来处理数据格式化（千位分隔符）和业务逻辑（期趸数据），这引入了文件系统依赖、清理复杂性、性能开销和潜在的数据陈旧问题。用户反馈表明临时文件机制可能导致数据不一致和系统不稳定。

## Goals / Non-Goals
- **Goals**:
  - 完全消除临时文件生成和清理机制
  - 将所有数据处理转换为内存操作
  - 提升系统响应速度和可靠性
  - 简化系统架构和维护复杂度
  - 确保数据始终使用最新版本

- **Non-Goals**:
  - 不改变任何用户可见功能
  - 不修改API接口（除了内部实现）
  - 不改变数据格式或业务逻辑

## Decisions

### Decision 1: 内存数据处理架构
**What**: 将千位分隔符格式化和期趸数据生成转换为内存中的DataFrame操作
**Why**:
- pandas DataFrame本身支持高效的内存操作
- 避免文件I/O开销（通常需要50-200ms）
- 消除临时文件权限和清理问题
- 确保数据操作的一致性

**Alternatives considered**:
- 继续使用临时文件但优化清理机制：仍存在根本的文件系统依赖问题
- 使用数据库临时表：引入新的复杂性和依赖

### Decision 2: 内存管理策略
**What**: 利用现有的 `MemoryDataManager` 来扩展内存数据处理能力
**Why**:
- 项目已有内存管理基础设施
- 避免重复造轮子
- 保持架构一致性

**Alternatives considered**:
- 创建新的内存处理模块：增加代码复杂度
- 直接在DataReader中处理：违反单一职责原则

### Decision 3: 渐进式迁移
**What**: 先实现内存处理功能，再移除临时文件代码
**Why**:
- 降低风险，确保每步都可测试
- 保持系统在迁移过程中的稳定性
- 便于回滚和问题定位

## Risks / Trade-offs

### Risk 1: 内存使用增加
- **Risk**: 内存中处理大数据集可能增加内存消耗
- **Mitigation**:
  - 实现内存使用监控
  - 对于超大文件，可考虑流式处理
  - 添加内存不足时的优雅降级机制

### Risk 2: 现有代码依赖
- **Risk**: GUI或其他组件可能依赖临时文件路径
- **Mitigation**:
  - 仔细搜索和更新所有临时文件路径引用
  - 保持API接口兼容性
  - 全面测试所有功能路径

### Trade-off: 复杂性转移
- **From**: 文件系统复杂性（权限、清理、路径）
- **To**: 内存管理复杂性（生命周期、引用）
- **Justification**: 内存管理更容易控制和测试，且性能更好

## Migration Plan

### Phase 1: 实现内存数据处理功能
1. 在 `MemoryDataManager` 中添加千位分隔符格式化方法
2. 在 `MemoryDataManager` 中添加期趸数据生成方法
3. 创建单元测试验证内存处理功能

### Phase 2: 更新调用方
1. 更新 `DataReader` 使用内存处理而非临时文件
2. 更新 `PPTGenerator` 直接使用内存数据
3. 更新 GUI 移除临时文件路径处理

### Phase 3: 清理临时文件代码
1. 移除所有临时文件创建方法
2. 移除临时文件管理类
3. 移除清理机制和相关配置
4. 更新文档和注释

### Phase 4: 测试和验证
1. 运行完整的功能测试套件
2. 性能基准测试对比
3. 内存使用情况监控
4. 用户验收测试
